## 1. KNOW-INSIGHT-01. "시스템이 본 것"을 "사용자가 본 것"으로 번역

**문제**:
- 시스템 내부에서는 layers=['phonological','semantic'], max_score=0.55 등 정확한 정보 보유
- 그러나 사용자 카드에서는 단순 숫자·아이콘 나열로 의미가 전달되지 않음
- 클로드 챗(다른 세션)은 같은 데이터를 보고 "Dual match 강신호 / 사명 매칭 부작용" 같은 통찰을 한 줄로 표현 가능

**해결**:
- 데이터 → 통찰 변환 계층 (`computeInsight`) 도입
- 모든 변환 규칙을 한 함수에 집중 (재사용 가능)
- 카드 UI는 변환된 통찰만 표시 (원시 점수는 보조 정보로)

**노하우**:
- "데이터를 보여주는 것"과 "데이터의 의미를 보여주는 것"은 별개의 UX
- ML/통계 시스템 UI 설계 시 항상 "이 숫자가 사용자에게 어떻게 들리는가?" 검증
- 차트·표·숫자보다 "한 줄 평가"가 직관 전달에 우수

## 2. KNOW-INSIGHT-02. 부작용 자동 감지 패턴

**문제**:
- 사피엔반도체 = 사명에 "반도체" 직접 포함 → semantic 매칭이 자동 발생
- 그러나 실제 사업은 DDI(디스플레이 구동 IC), 메모리 슈퍼사이클과 무관
- 사용자가 매수 결정 시 이 부작용을 모르면 잘못된 신호로 진입 가능

**해결**:
- THEME_KEYWORDS_FOR_WARN 사전 (반도체·바이오·제약·테크·로보틱스 등) 구축
- 종목명에 테마 키워드가 들어있고 + semantic 매칭이 그 테마로 발생한 경우 → 경고
- "사명 매칭 (XX 박스 부작용 가능)" 메시지

**노하우**:
- 자동 매칭의 가장 위험한 부작용은 "이름만 비슷한 종목"
- 단타 시스템 = 패턴 매칭 시스템 → 부작용 명시가 시스템 신뢰도의 핵심
- 향후 확장: KRX 공식 섹터/테마 분류와 sanity check (예: "반도체" 사명 + KRX 섹터="반도체" 일치 → 경고 OFF)

## 3. KNOW-UX-01. 별점 vs 숫자 — 둘 다 보여주기

**문제**:
- max_score 0.55는 정확한 정보지만 직관 X
- "★★★★/4"가 즉시 강도 인식 가능
- 그러나 별점만 보여주면 정확한 수치 손실 → 신뢰도 비교 시 불편

**해결**:
- 별점(시각화) + 숫자(정확값)를 함께 표시
- 별점은 호버 시 "max ${score}" 툴팁

**노하우**:
- 정량 데이터의 UX = "직관" + "정확" 둘 다 필요
- 별점만 → 정확성 손실
- 숫자만 → 직관 손실
- 두 layer 동시 표시가 정답

## 5. KNOW-VERIFY-01. 카드 변경의 cross-validation 방식

**상황**: 새 UI 컴포넌트 추가 시 어떻게 사전 검증할까?

**접근**:
1. **JS 헬퍼 함수를 분리 가능하게 작성** → Node로 실행 가능
2. 실 데이터(matches_*.json) 입력
3. 핵심 종목의 출력 검증 (사피엔반도체, 아이진, 차백신연구소 등)
4. 알려진 정답(클로드 챗 평가)과 cross-check

**노하우**:
- UI 변경도 unit test 가능 — JS 헬퍼 분리가 핵심
- 브라우저 E2E는 사용자 확인에 맡기되, 알고리즘은 자동 검증
- 실 데이터 + 알려진 정답 = best validation
