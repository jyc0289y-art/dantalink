## KNOW-DATA-SOURCE-01. KRX-DESC 발견의 가치

**노하우**:

## KNOW-AUTH-01. 5단계 authenticity 분류의 균형

**문제**: 단순 "사명 매칭 vs 진짜 매칭" 이분법은 거짓 양성/음성 다수.

**5단계 분류**:
1. `phonological` — Layer A 음운, auth 평가 대상 아님
2. `genuine` — 키워드가 Industry/Products에 실제 포함
3. `name_only` — 사명에만 있고 사업과 무관
4. `standard` — 어디에도 명시 없음 (약한 시그널)
5. `unknown` — 섹터 정보 미등록

**노하우**:
- "사명 vs 사업" 외에 "전혀 없음" (standard)도 별도 분류 필요
- "데이터 없음" (unknown)을 false-positive로 처리하지 않음 — graceful degradation

## TRAP-MATCHTYPE-AGGREGATION-01. 종목 전체 matchType 산출 로직

**상황**: 종목당 매칭 N건. 각각 authenticity 다름. 종목 종합 matchType은?

**시도 1 (실패)**: 다수결 — 가장 많은 auth가 종목 matchType
- 문제: 카테고리 테마 매칭이 많으면 standard 다수 → genuine 1건 묻힘

**시도 2 (채택)**:
- semanticTotal = 0 (음운만): `phonological`
- name_only > 0 & genuine == 0: `name_only` (모든 의미 매칭이 부작용)
- genuine > 0 & name_only == 0: `genuine` (모든 의미 매칭이 진짜)
- genuine > 0 & name_only > 0: `mixed` (혼재)
- 그 외: `standard`

**노하우**:
- 집계 로직은 "최악의 경우"가 우선 (NAME-ONLY 1건이라도 있으면 mixed로 강등)
- "최고의 경우"만 보면 부작용 놓침
- 단타 매수 결정 = 보수적 집계 + 사용자가 직접 검증

## KNOW-LAZY-LOAD-01. 비동기 데이터의 재렌더 트리거

**상황**: KRXSectorDB는 fetch로 비동기 로드. 페이지 첫 렌더 시 아직 로드 안 됨 → matchType="unknown"으로 표시. 로드 완료 후엔 정확한 분류 필요.

**구현**:
```javascript
(async () => {
  if (window.DantaLink && window.DantaLink.KRXSectorDB) {
    STATE.sectorDB = await window.DantaLink.KRXSectorDB.fromUrl();
    // 섹터 DB 로드 완료 후 현재 화면 다시 렌더 (matchType 반영)
    if (STATE.data) render();
  }
})();
```

**노하우**:
- 비동기 데이터 의존 컴포넌트는 "로드 전 / 로드 후" 2번 렌더 패턴
- 로드 전: graceful degradation (matchType "unknown", 배지 숨김)
- 로드 후: 강제 재렌더 (사용자 인터랙션 없이도)
- 사용자가 빠른 클릭 시 변경된 UI에 혼란 — fade 효과 검토

## KNOW-ALGORITHM-LIMIT-01. 단순 substring의 미세 한계

**관찰**: "로보틱스" 키워드 매칭 → 엔젤로보틱스 = NAME-ONLY로 분류됨
- 종목명: 엔젤**로보틱스** ✓
- Products: "재활 전문 웨어러블 **로봇**…" — "로보틱스" 정확히 없음

**원인**: substring 매칭이라 "로봇" ≠ "로보틱스"

**노하우**:
- "기술적 정확"과 "의미적 정확"은 다름
- 한국어 키워드의 어미 변화 (로봇/로보틱스/로보트), 유사어 (AI/인공지능) 처리는 동의어 사전 필요
- 향후 개선: KEYWORD_SYNONYMS 사전 — `{'로보틱스': ['로봇','로보트'], 'AI': ['인공지능']}`
- 또는 임베딩 기반 유사도 (LLM)
