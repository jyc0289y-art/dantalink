## Part 1. 환경 관련 시행착오

### TRAP-WIN-01. iCloud 동기화 placeholder
**문제**:
- 프로젝트 폴더 `docs/`가 윈도우 탐색기에서 빈 폴더로 보임
- 메타데이터(폴더 생성 시각)는 동기화됐지만, 파일 본체는 placeholder만 (iCloud "online-only")
- 사용자 메시지: "docs에 문서가 없는데"

**원인**:
- iCloud for Windows의 "Files On-Demand" 정책: 메타데이터 우선, 본체는 접근 시 다운로드
- 본 세션은 윈도우에서 docs/ 본체에 한 번도 접근 안 했음 → 다운로드 미트리거
- PowerShell `Get-ChildItem`도 placeholder는 안 보임 (Length=0인 항목이 어쩌다 보일 뿐)

**우회 기법**:
1. `dantalink.zip` (사용자가 클로드 챗에 보낸 검토용 zip)이 같이 동기화되어 있음을 활용
2. zip 내용을 임시 디렉토리에 추출 → 한글 파일명 인코딩 깨짐 (cp949 vs UTF-8 충돌)
3. zip 항목 인덱스(`zip.Entries[27]` 등)로 접근하여 인코딩 무관 추출
4. 안전한 파일명(`doc_27_collector_gui.html`, `doc_28_visualization.html`, ...)으로 추출
5. 사용자 요청 시 한글 원본 이름(`DantaLink_데이터수집기.html` 등)으로 PowerShell Rename-Item
6. Copy-Item으로 프로젝트 `docs/`에 영구 복원

**노하우**:
- iCloud + Windows 환경에서는 `Get-ChildItem -Force`로 placeholder 식별 가능 (Attributes에 `Offline`)
- zip 내 한글 파일명을 다루려면 PowerShell `[System.IO.Compression.ZipFile]::OpenRead()` 후 `$zip.Entries[i].Open()`으로 스트림 추출
- 새 파일 생성 후 iCloud 동기화 확인: `Get-ItemProperty`의 `LastWriteTime` vs Length 확인

### TRAP-WIN-02. 콘솔 한글/유니코드 깨짐
**문제**:
- Bash 출력의 한글이 `�` 또는 `?`로 깨져 표시
- Python CLI 실행 시 `UnicodeEncodeError: 'cp949' codec can't encode character '✓'` 크래시
- 코드 자체는 UTF-8 정상, 콘솔만 깨짐

**원인**:
- 윈도우 콘솔 기본 코드 페이지: cp949 (한국어 윈도우)
- Python `sys.stdout.encoding == 'cp949'`
- ✓(✓), 💾(💾), ▶(▶) 등 BMP 외 유니코드 출력 시 cp949가 인코딩 거부 → 예외

**해결 (세 가지 동시 적용)**:
1. **cli.py main() 시작에 stdout/stderr reconfigure** (영구):
   ```python
   if sys.stdout.encoding and sys.stdout.encoding.lower() != "utf-8":
       try:
           sys.stdout.reconfigure(encoding="utf-8", errors="replace")
           sys.stderr.reconfigure(encoding="utf-8", errors="replace")
       except (AttributeError, ValueError):
           pass
   ```
2. **환경변수 PYTHONIOENCODING** (런타임):
   ```bash
   PYTHONIOENCODING=utf-8 python -m dantalink.cli scan ...
   ```
3. **콘솔 코드 페이지 강제** (선택, 영구):
   ```cmd
   chcp 65001
   ```

**노하우**:
- `errors="replace"`로 설정하면 인코딩 불가 문자가 `?`로 치환되어 크래시 회피
- pytest는 자체 stdout 캡처로 영향 없음 (테스트 결과는 정확)
- PowerShell vs cmd.exe: PowerShell 7+는 UTF-8 더 친화적, cmd는 chcp 필수
- VS Code 통합 터미널은 기본 UTF-8 (윈도우에서도) → 개발 시 편리

## Part 2. 외부 API 시행착오

### TRAP-API-01. GDELT Rate Limit IP 차단
**문제**:
- 첫 라이브 수집 시 4개 키워드 순차 호출 → **429 Too Many Requests** 적중
- 자체 검토 보고서(M2)에서 미리 예측한 이슈가 실제로 발생

**진단 과정**:
1. 처음엔 단순 일시 차단으로 추정 → 2초 sleep 추가 (M2 일부 미리 적용)
2. 60초 대기 후 단일 키워드 재시도 → **여전히 429**
3. 직접 httpx GET 호출 → 응답 body에 명시:
   ```
   Please limit requests to one every 5 seconds or
   contact kalev.leetaru5@gmail.com for larger queries.
   ```

**원인**:
- GDELT 공식 한도: **1 request / 5 seconds** (서버 응답 body에 명시되지만 공식 문서엔 모호)
- 우리는 첫 시도에 4개 키워드 거의 동시 호출 → 분당 12회 → 한도 초과
- 한 번 차단되면 ~10분 이상 cooldown 지속 (실측)

**해결**:
1. `GDELT_INTER_QUERY_SLEEP_SECONDS = 5.5` (안전 마진 0.5초)
2. `User-Agent: DantaLink/0.1 (+research)` 헤더 명시 (익명 클라이언트는 더 빠르게 차단)
3. 향후: HTTP 429 시 exponential backoff 추가 (M2 완성형)

**노하우**:
- 외부 API 호출 전 응답 body 직접 확인하면 미공개 한도 발견 가능 (`r.text`)
- "공개 API" 라도 익명 호출은 식별 가능한 User-Agent 권장
- 윈도우 IP에서 차단되면 맥북·모바일 핫스팟 등 IP 변경으로 우회 가능
- 진짜 대량 호출 필요시 GDELT 운영자(kalev.leetaru5@gmail.com)에게 직접 요청

## Part 3. 매칭 로직 시행착오

### TRAP-MATCH-01. regex `{2,5}` 으로 1자 토큰 누락 → 샘표 매칭 실패
**문제**:
- 초기 phonological matcher: `re.findall(r"[가-힣]{2,5}", text)`
- "샘 알트먼" → "샘"(1자) 누락, "알트먼"(3자)만 추출
- 결과: 샘표(007540) 매칭 실패
- 방법론 §2.4의 핵심 사례 데모 실패

**해결**:
1. regex를 `{1,5}` 로 변경
2. 1자 stopword 사전 추가 (`_STOPWORDS_SINGLE`):
   ```python
   _STOPWORDS_SINGLE = {"이", "그", "저", "또", "다", "두", "세", "네", "큰", ...}
   ```
3. 1자 토큰에는 score × 0.7 페널티 부여 (false positive 위험 보상)
4. seen_first 중복 체크로 같은 첫 음절의 다른 토큰 무시

**노하우**:
- 외국 인명 한국어 표기는 보통 "1-2자 한글 + 공백 + 2-4자 한글" 패턴 ("샘 알트먼", "팀 쿡")
- 단순 1자 허용 시 한국어 조사·접속사 다수 매칭 → stopword 필수
- 향후 정교화 옵션: 외국 인명 패턴 전용 regex (`[가-힣]{1,2}\s+[가-힣]{2,4}`)
- KoNLPy + NER 라이브러리 도입 시 더 정확하지만 의존성 무거움

### TRAP-MATCH-02. `if token == s.name: continue` 의 의도와 실제 효과 불일치
**의도된 동작** (코드 작성 시):
- 트리거에 "삼성전자" 직접 언급 시 Layer B에서 잡으므로 Layer A 중복 회피

**실제 발생한 문제**:
- "아이진 백신주 급등" 트리거 → "아이진" 토큰
- KRX 종목명 "아이진"(185490)과 정확 일치
- 스킵 로직으로 인해 **EXACT_MATCH (score 0.55)가 매칭에서 제거됨**
- 결과: 아이진이 Top 30에 안 들어옴 → 방법론 §2.4 "카테고리 매핑" 사례 데모 실패

**해결**:
- 해당 라인 제거
- exact match는 Layer A의 가장 강한 신호로 살림
- 결과: 아이진(185490) **1위 등극** (score 0.55) ✅

**노하우**:
- "중복 회피" 의도의 스킵 로직은 정작 가장 강한 신호를 차단할 위험
- Layer A(음운)와 Layer B(의미)는 서로 보완적이지 경쟁적이지 않음 → 같은 종목이 두 레이어에서 잡혀도 OK (corroboration)
- "exact match는 음운+의미 동시 매칭급"이라는 방법론 §4 Stage 4의 점수 체계와도 일치

### TRAP-MATCH-03. 시총 정렬 기반 max_per_token으로 인한 누락
**문제**:
- 처음: `stocks_sorted = sorted(stocks, key=lambda s: s.marcap_eok, reverse=True)`
- "아"로 시작 종목 중 시총 큰 종목부터 max_per_token=3개 → 아이센스(4,987억), 아세아(4,632억), 아모레퍼시픽우(4,250억) 선택
- 아이진(185490, 시총 966억) → 41번째 → 잘림

**해결**:
- 정렬 키를 (점수 내림차순, 시총 내림차순) 튜플로 변경
- exact (0.55) > first2 (0.40) > first1 (0.30) 자동 우선
- 동점일 때만 시총 큰 종목 우선
- 코드:
  ```python
  scored.sort(key=lambda x: (x[0], x[2].marcap_eok or 0), reverse=True)
  ```

**노하우**:
- 점수 기반 정렬이 시총 정렬보다 우선
- max_per_token 한도가 있을 때 정렬이 결과의 정확도를 좌우
- 매칭 점수 = 정답률 직결, 시총은 시그널 강도와 무관 (오히려 단타에서는 작을수록 좋음)

### TRAP-MATCH-04. total_score 정렬의 함정 — 카테고리 매칭 폭주
**문제**:
- 처음 CLI 출력: total_score(누적) 기준 정렬
- RSS 20건 매칭 결과 상위 20개가 모두 반도체/테크 대형주
- 한미반도체(27조), LG에너지솔루션(89조) 등 — 단타와 무관
- 점수가 거의 동일 (5.60~7.10) → 분별 어려움

**원인**:
- AI 키워드가 14개 이벤트에 등장 → 분류기가 14건을 TECH로 분류
- Layer B의 "반도체" 테마 키워드 → 매일 14건 누적 매칭
- 한 종목당 14회 × 0.40점 = 5.6점 누적
- 단일 매칭은 약하지만 총합은 큼

**해결**:
- `max_score` (단일 매칭 최고 점수) 추가 추적
- 정렬 기본값을 `max_score` 우선으로 변경
- `--sort` 옵션으로 max/total 선택 가능
- 결과: 비상교육(0.55 단일) > 사피엔반도체(0.50 × 17회 누적) 정렬 변경

**노하우**:
- 누적 점수(total)는 강도가 아닌 빈도 → 노이즈
- 최고 점수(max)는 한 트리거의 진짜 강도 → 시그널
- 단타에서는 max 우선이 자연스러움
- 다중 매칭은 보너스로 표시(매칭 수 컬럼)하되 정렬엔 보조

### TRAP-MATCH-05. Stopword 부족으로 noise token 폭주
**문제 사례**:
- "선포" → 선익시스템(171090), 선진(136490), 선도전기(007610) 등 매칭 (의미 없음)
- "고조" → 고려신용정보(049720) 매칭
- "모색" → 모베이스전자(012860) 매칭
- "협력" → 한국협력... 등 매칭

**해결**:

- 추가된 stopword: 선포, 모색, 고조, 긴장, 충돌, 급등, 급락, 출범, 선출, 관측, 주장, 강조, 예상, 계획, 추진, 체결, 협상, 검토, 검찰, 지지, 반대, 수상, 포착, 발생, 기업과, 한국, 국내, 해외, 글로벌

**노하우**:
- 한국어 자주 쓰는 동사/명사/형용사는 stopword 필요
- 향후 형태소 분석기(KoNLPy)로 자동 추출 가능
- 가장 정교한 해법: BERT/Claude 임베딩 기반 의미 유사도 (현재는 단순 단어 매칭)

### TRAP-MATCH-06. 카테고리 테마 키워드의 폭주
**문제**:
- HEALTH 카테고리 매핑: `["바이오", "제약", "백신", "메디", "팜", "셀", "헬스", "한미", "녹십자"]`
- 9개 키워드 × 종목 substring 매칭 → 한 이벤트당 50+ 종목 매칭
- 모두 score 0.40으로 동일 → 순위 무의미

**부분 해결 (현재)**:
- max_per_keyword=5 한도로 시총 큰 종목만 5개
- 향후 키워드별 가중치 차등화 필요

**향후 정교화 방향**:

## Part 5. 코딩·아키텍처 노하우

### KNOW-CODE-01. backward compat을 위한 deprecated alias 패턴
**적용**: `KRXCollector` → `DARTCollector` 변경 시
```python
class KRXCollector(DARTCollector):
    """[Deprecated] DARTCollector의 별칭."""

    def __init__(self, *args, **kwargs):
        warnings.warn(
            "KRXCollector is deprecated; use DARTCollector instead.",
            DeprecationWarning,
            stacklevel=2,
        )
        super().__init__(*args, **kwargs)
```

**장점**:
- 모듈 import 시점이 아닌 인스턴스화 시점에만 warning (스팸 X)
- subclass라 모든 기능 자동 상속
- 기존 코드 (외부 사용자) 한 줄도 안 깨짐
- `c.source` 자동으로 `EventSource.DART` 반환 (자식 클래스에서 클래스 변수 상속)

**노하우**:
- DeprecationWarning은 default로 숨겨짐 (`python -W default` 또는 pytest는 자동 표시)
- `stacklevel=2` 중요 — 호출자 위치를 가리켜야 디버깅 편의
- 향후 v1.0 정도에서 alias 완전 제거 + breaking change 명시

### KNOW-CODE-02. dataclass + field(default_factory=...)
**적용**: `TriggerEvent`, `MatchCandidate`
```python
@dataclass
class MatchCandidate:
    stock_code: str
    stock_name: str
    ...
    details: dict[str, Any] = field(default_factory=dict)
```

**노하우**:
- 가변 기본값(list, dict)에는 반드시 `field(default_factory=...)` 사용
- 직접 `details: dict = {}` 쓰면 모든 인스턴스가 같은 dict 공유 (악명 높은 함정)
- `asdict()`로 JSON 직렬화 가능 (다만 datetime은 별도 처리 필요)

### KNOW-CODE-03. from __future__ import annotations
**적용**: 모든 모듈
```python
from __future__ import annotations
```

**효과**:
- 모든 타입 힌트가 string으로 평가됨 (PEP 563)
- Python 3.10+ 신문법 (`list[str]`, `X | None`) 사용 가능
- forward reference 자동 처리
- 런타임 비용 감소

**노하우**:
- `from typing import Optional, List` 같은 import 줄어듦
- `Optional[str]` 대신 `str | None` 가능
- 단, runtime introspection 필요 시 (`get_type_hints(...)`) 명시적 평가 필요

### KNOW-CODE-04. ABC + safe_method 패턴
**적용**: `BaseCollector.safe_fetch`, `BaseMatcher.safe_match`
```python
class BaseCollector(ABC):
    @abstractmethod
    async def fetch(self) -> list[TriggerEvent]: ...

    async def safe_fetch(self) -> list[TriggerEvent]:
        try:
            return await self.fetch()
        except Exception as e:
            self.logger.exception(...)
            return []
```

**효과**:
- 구현체는 fetch만 작성 (에러 처리 내장)
- 호출자(pipeline)는 safe_fetch만 부르면 됨
- 단일 실패가 전체 파이프라인 막지 않음

**노하우**:
- "fail-safe" vs "fail-fast" 선택은 도메인 의존적
- 데이터 수집은 fail-safe (한 소스 실패 OK)
- 결제·돈 관련은 fail-fast (실패 시 명시적 중단)

### KNOW-CODE-05. 인덱스 자료구조 (defaultdict 활용)
**적용**: `KRXStockDB`
```python
self.by_first_syllable: dict[str, list[KRXStock]] = defaultdict(list)
self.by_substring: dict[str, list[KRXStock]] = defaultdict(list)

for s in stocks:
    self.by_first_syllable[s.syllables[0]].append(s)
    for i in range(len(s.name) - 1):
        bigram = s.name[i:i + 2]
        self.by_substring[bigram].append(s)
```

**효과**:
- O(1) 조회 (2,878 종목 → 330 첫음절 + 4,461 bigram 인덱스)
- 메모리 ~ 1MB 미만
- 매번 전체 스캔(2,878회) 대비 ~100배 빠름

**노하우**:
- bigram 인덱스는 substring 검색 가속에 효과적 (suffix tree 없이도 충분)
- 3+ 글자 검색은 bigram 후보 → 정확 매칭 2단계
- 인덱스 갱신은 DB 로드 시 1회 + 영속화 시 캐시 무효화

### KNOW-CODE-06. CSV 인코딩 — UTF-8 with BOM
**적용**: `data/krx_stocks.csv`
```python
df.to_csv(out, index=False, encoding='utf-8-sig')
# 로드 시:
with open(csv_path, "r", encoding="utf-8-sig", newline="") as f:
    reader = csv.DictReader(f)
```

**효과**:
- Excel에서 한글 정상 표시 (BOM이 자동 인코딩 감지)
- Python csv 모듈은 BOM 자동 제거 (`utf-8-sig`)

**노하우**:
- `utf-8` vs `utf-8-sig` 차이: 전자는 BOM 없음 (Linux/macOS 표준), 후자는 BOM 있음 (Windows/Excel 호환)
- 사용자가 Excel 직접 열어볼 가능성 있으면 `utf-8-sig` 권장
- 다만 JSON은 BOM 금지 (RFC 8259) → JSON은 `utf-8`
