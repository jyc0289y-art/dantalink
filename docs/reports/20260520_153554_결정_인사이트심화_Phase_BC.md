## 2. 사명/실사업 구분 — 옵션 비교 (핵심 결정)

### 문제 정의
사피엔반도체(452430): 사명에 "반도체" 들어가서 매칭되지만 **실제 사업은 DDI (디스플레이 구동 IC)** — 메모리/HBM 모멘텀과 무관.
현재 시스템은 사명 매칭만 보고 매칭하므로 **"가짜 카테고리 매핑"을 부작용 경고로 표시**만 함.
사용자는 **"진짜 매칭 vs 가짜 매칭"이 자동 구분되길** 원함.

### 4가지 옵션

| 옵션 | 설명 | 정확도 | 구현 비용 | 운영 부담 |
|------|------|------|---------|---------|
| **A** | 현재 상태 유지 — 단순 경고만 ("⚠ 사명 매칭 가능") | ★★ | 0 | 0 |
| **B** | **사전 큐레이션 사전** — `data/stock_actual_business.json` 수동 정리 (예: 사피엔반도체 → "DDI, 디스플레이") | ★★★★ | 중간 (사전 작성) | 중간 (수동 업데이트) |
| **C** | KRX 공식 섹터/테마 데이터 추가 — finance-datareader로 다운로드 + cross-check | ★★★★★ | 큼 (데이터 추가 + 로직) | 작음 (자동 갱신) |
| **D** | LLM 호출 — 종목명 + 사업 영역 매칭 검증 | ★★★★★ | 큼 (API 통합) | 비용·지연 |

### 권장안: **B + C 하이브리드 — C 우선, B는 fallback**

**근거**:
- C가 가장 정확 + 자동화 (finance-datareader가 sector 정보 제공 가능)
- 데이터 다운로드 가능: `fdr.StockListing('KRX-DESC')` 또는 KRX OpenAPI 시도 필요
- 만약 sector 정보 어렵다면 B로 fallback (5-10개 알려진 부작용 사례 수동)
- D(LLM)는 비용·외부 의존이라 보류

## 3. Phase B 오버레이 상세창

### 형태
- 매칭 카드(우측 패널의 Match Evidence) 클릭 → 화면 중앙 모달
- 모달 내용:
  - 트리거 이벤트 제목 + 본문 (events_*.json의 body 사용)
  - 원본 URL (클릭 시 새 탭)
  - 카테고리·키워드·발행 시각
  - 매칭 메커니즘 + 점수
- 닫기: ✕ 버튼 / 오버레이 밖 클릭 / Esc 키

### 데이터 소스
- 현재 matches_*.json의 matches[]는 event_title, event_category, layer, score, matched_on만 보유
- 원본 body·url·published_at 부재 → events_*.json도 같이 로드 + event_id로 join
- 또는 matches.json 생성 시 원본 데이터 inline embed (Python CLI 수정)

→ **클라이언트 사이드 join**이 가벼움. Stock Match GUI 로드 시 events_*.json도 fetch.
