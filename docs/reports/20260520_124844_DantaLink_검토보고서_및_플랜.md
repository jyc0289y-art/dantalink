# DantaLink 프로젝트 상세 검토 보고서

## Context

사용자(진영)가 맥북 없이 윈도우 PC에서 임시로 DantaLink 프로젝트를 검토 요청. 본격 작업은 맥북에서 진행할 예정. 클로드 챗에서 검토한 결과물이 `dantalink.zip`(66KB, 2026-05-20 11:36 생성)으로 패키징되어 프로젝트 폴더에 위치. 실제 `docs/` 폴더는 iCloud 동기화가 되지 않아 비어 있어, zip에서 임시 디렉토리(`%TEMP%\dantalink_review_20260520114357\`)로 추출하여 검토함.

**목적**: Stage 01(데이터 수집기) 코드 + 4개 설계 문서 전체를 점검하고, Stage 02(매칭 후보 생성) 본격 진입 전 개선 우선순위를 도출해 맥북 세션이 즉시 이어받을 수 있도록 준비.

## 2. 종합 진단

### 2.1 ✅ 잘 된 점

1. **방법론 문서 학술적 견고함**
   - 카너먼·트버스키(1972, 1973), Rashes(2001), Da/Engelberts/Gao(2011), Barber & Odean(2008) 등 13개 핵심 논문 인용
   - 가설 → 이론 → 사례(아이진/샘표) → 공식(Kelly 1/4) → 운영 가이드 일관된 흐름
   - 인지 매칭 정교함 스펙트럼(펀더멘털 → 산업인과 → 카테고리 → 의미점화 → 이름유사성 → 티커혼동) 정리 우수

2. **코드 아키텍처 깔끔**
   - `BaseCollector` ABC + `safe_fetch` 패턴 → 단일 수집기 실패가 파이프라인 전체를 막지 않음 (`base.py:37-48`)
   - `TriggerEvent.make_id`로 source-prefix SHA256 → cross-source 중복 제거 (`models/__init__.py:72-79`)
   - asyncio 일관성: GDELT/DART는 httpx async, RSS는 `asyncio.to_thread`로 동기 라이브러리(feedparser) 우회 (`rss.py:77`)
   - `@dataclass` + type hints + `__future__.annotations` 일관 적용
   - 체이닝 가능한 fluent API: `pipeline.register(...).register(...).on_event(...)` (`pipeline.py:36-45`)

3. **iOS 통합 사전 설계**
   - KIS REST API (Base URL, OAuth, 주문 코드 TTTC0802U) 미리 조사 완료
   - Tier 1(자동) / Tier 2(반자동 텔레그램 승인) / Tier 3(외출 중 Shortcuts) 하이브리드 아키텍처 명확
   - JSON 직렬화 스키마(`to_dict()`)가 iOS 앱과 공유 가능하게 설계됨

4. **개인 리스크 의식**
   - 운영자 건강·의사결정 피로·약물 상호작용 등 운영 리스크에 포함 (구체 의료 정보는 internal 문서로 분리)
   - 권장 시작 순서(페이퍼 2주 → 자본 1~2% → 자동화 → 사람 in-the-loop) 보수적이고 합리적

5. **테스트 분류기 8건**
   - tie-break 우선순위 검증, word boundary("AI" vs "AID") 검증, 한/영 동시 매칭 case 커버
   - `make_id` 결정론·source separation 검증 등 모델 핵심 5건 보유

- **현상**: 클래스명·EventSource·CLI 옵션·디렉토리 어디서나 "KRX"인데 실제로는 **DART(금감원 전자공시) API** 호출 (`collectors/krx.py:23` `DART_BASE_URL`)

### 2.3 🔧 중기 개선 (Medium Priority — Day 2-3)

#### M1. 데이터 영속성(Persistence) 부재
- **현상**: 모든 수집 결과 메모리 + JSON 파일 옵션만. SQLite/PostgreSQL 통합 없음 (`pipeline.py:47-82`)
- **영향**: 24/7 데몬 운영(방법론.md 5.2 Mac Studio 시나리오) 시 재기동마다 같은 이벤트 재수집 → DART 일일 10,000건 한도 빠르게 소진, GDELT rate limit 적중
- **권장**: SQLite 기반 `events.db` 도입
  - `event_id` PRIMARY KEY로 cross-session dedup
  - Stage 02 매칭 결과(`match_candidates` 테이블)도 동일 DB에 join 가능하게 설계
  - SQLAlchemy 도입은 과잉, `sqlite3` 모듈 + 박막 wrapper로 충분
- **위치 후보**: `dantalink/storage/sqlite.py`

#### M2. Rate limiting / Retry 미구현
- **GDELT**: 키워드 N개에 대해 순차 호출(`gdelt.py:59-95`)이지만 rate limit 처리 없음. GDELT는 실측 분당 ~10 요청 제한 추정
- **DART**: 일 10,000건이지만 RPS 제한 없음 → spike 시 일시 차단 위험
- **권장**:
  - `asyncio.Semaphore(concurrency=2)` + 호출 간 `await asyncio.sleep(0.5)`
  - HTTP 5xx/429 시 exponential backoff (`httpx-retry` 또는 자체 구현 5줄)
  - `BaseCollector`에 공통 retry 헬퍼 추가

#### M3. classifier 키워드 사전 한계
- **단순 substring 매칭**의 false positive 우려:
  - "백신" 키워드가 "백신패스 폐지" 같은 정책 기사에도 매칭 → HEALTH로 잘못 분류
  - "AI" word boundary로 보호되지만 한글 "AI"는 임의 위치 매칭 (예: "DAILY")
- **tie-break 우선순위 재고**:
  - 현재 `HEALTH > GEOPOLITICAL > TECH > DISASTER > REGULATORY > PERSON` (`classifier/__init__.py:72-79`)
  - 진영님 사례 중 **샘표(PERSON 매칭)** 가 가장 강력한 단타 트리거였는데, PERSON이 최하위 우선순위
  - 음운 매칭의 시작점인 PERSON 카테고리가 약화되면 Stage 02 Layer A의 핵심 트리거가 손실
  - **권장**: PERSON을 GEOPOLITICAL과 TECH 사이로 상향, 또는 tie-break 자체를 신뢰도(매칭 키워드 수 + 카테고리 가중치) 기반 점수로 전환

#### M4. GDELT 한국어 검색 비효율
- **현상**: 코멘트(`gdelt.py:54`)에 "GDELT는 영어 검색이 강하므로 영문 키워드 위주"라 명시했지만, `cmd_scan`에서는 모든 키워드(한국어 포함)를 GDELT에 전달
- **결과**: 한국어 키워드에 대한 GDELT 응답이 매번 거의 빈 배열 → API 호출 낭비
- **권장**: 키워드별 source routing
  ```python
  english_kw = [k for k in keywords if all(ord(c) < 128 for c in k)]
  korean_kw = [k for k in keywords if k not in english_kw]
  # GDELT에는 english_kw만, RSS/DART/Telegram에는 모두 전달
  ```

#### M5. 다중 소스 신뢰도 가중치 없음
- **현상**: `pipeline.run_once`(`pipeline.py:62-69`)는 단순 dedup만. 동일 사건이 GDELT+RSS+Telegram 4곳에서 잡혀도 1건 처리
- **잠재 가치**: 다중 소스 corroboration은 신뢰도의 강력한 신호 (특히 텔레그램 단독은 노이즈 많음 → RSS+GDELT 동시 등장 시 신뢰도↑)
- **권장**: dedup 시 첫 등장 이벤트의 `raw_payload`에 `corroboration: [source1, source2, ...]` 누적. Stage 02 신뢰도 점수에 반영

#### M6. 백테스트 인프라
- 방법론 문서가 자체 인지: "백테스트 한계 — 과거 음운 매칭 데이터셋 부재" (7.1 시스템 리스크)
- **권장**:
  - 페이퍼 트레이딩 2주 동안 수집된 모든 TriggerEvent + Stage 02 매칭 후보 + 매칭 종목 historical price(D~D+5)를 자동 저장
  - 사후 분석 스크립트: 매칭 후보 종목이 D+1~D+5에 어떻게 움직였는지 통계
  - Data Source: KRX OpenAPI(KRX statistics) 또는 한국투자증권 historical price API
